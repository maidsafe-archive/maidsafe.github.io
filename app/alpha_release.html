SAFE Network Alpha-1

We are delighted to announce the immediate availability of the Alpha release of the SAFE Network, which incorporates user hosted network nodes (Vaults), client nodes (Launcher) and some initial demo client applications.

This release enables users out with MaidSafe to contribute their resources to the network for the first time and is an important first step into a new paradigm of secure, private and free range of Internet services. These services initially include storage of private data, creation of public names and services (such as web sites) and a very basic Network File System.

Our roadmap defines up and coming network features.

What is it?

This is the Alpha-1 release of the SAFE Network, a network that connects users computers together to create an alternative to a server based internet infrastructure.

This release is available for 64 bit desktop versions of Windows, Mac and Linux and includes the following items:

Vault (Network node)

When you download something from the SAFE Network, the data comes from a collection of many distributed vaults.

  • Runs in the background.
  • Contributes resources to the Network.

Launcher (Client node)

Launcher is the way you connect to the SAFE Network. You use the Launcher to login securely to the network and to authenticate our apps.

  • Create credentials and login without the knowledge or permission of any third party - Self Authentication.
  • Authenticates your apps and allows them to connect to the SAFE Network.

Demo App

The Demo app allows you to try out the core features.

  • Create your public identity.
  • Store and manage your public data.
  • Store and retrieve your private data.

IMPORTANT:

Please do not run a vault unless you are happy with this temporary limitation.

Vaults will log their information to a server to allow measurement of the core algorithms. We have attempted to ensure that no personal information will be logged, but advise users to assume everything is logged. This information will be used by the development team to track and improve network performance during this phase.

Objectives

The purpose of this release is to validate the code in a real world setting. Helping us establish if the network enables connections via certain types of routers (NAT Traversal), identify bugs and generally obtain feedback that helps us improve the user experience within future releases.

What is an Alpha release?

IMPORTANT:

This software should be considered as an Alpha release meaning that the software will be unstable and will cause data loss.

During the course of making improvements, MaidSafe will need to restart the network from time to time and this will result in the loss of all data and registered public identities. For clarity, MaidSafe cannot remotely stop user Vaults, rather we will periodically release new Vaults that will form a new network.

This software does not contain all the features that we envisage during future Alpha and Beta releases, the implementation of safecoin and messaging will be added in due course and will be communicated the development roadmap.

Limitations

As an Alpha this release has some technical limitations (listed below) that require the user to be either directly connected, or forward a TCP port and specify that in the Crust config file. This Alpha is therefore aimed at technical users.

  • To run a vault, it is currently necessary to be either directly connected or forward a TCP port and specify that in the Crust config file.
  • File size will currently be limited to 5Mb.
  • Vault size will currently be limited to 30 MB (30 chunks).
  • Storage per client account is limited to 50 MB (50 chunks).
  • Churn handling - replacing connections and relocating data after nodes joined or left the network - is still very rudimentary. It will almost certainly fail if there are hundreds of megabytes of data to relocate, which is why we set a low Vault size limit for now and ignore the one set in the Vault config file.
  • Zombie data: If there is heavy churn it can happen that a client requests deletion of data while that same data is being relocated. In that case, sometimes the data comes back from the dead after its deletion.
  • Known issue - occasionally the client may not receive enough “agreeing” responses from a network operation in order to decide the result. This causes the client to hang. It should just be restarted if this happens.
  • There is still a lot of work to be done in the Routing library for it to make the security guarantees needed not only for safecoin but also for preventing anyone from taking over the network entirely. There are a lot of known attacks that the network is vulnerable to.
